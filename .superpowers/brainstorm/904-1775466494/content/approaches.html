Three approaches to improving the app

Pick the one that feels right — I'll recommend my favourite below.

A

Foundation First ⭐ Recommended

Fix data transparency (sync status + last-refreshed indicators) and code quality (error handling, API layer) first. Then build matchup intelligence on top of a solid, reliable base. Slower to get new features but everything works properly when you get there.

Pros

  • No silent failures
  • Trustworthy data before showing analysis
  • Easier to debug matchup feature later

Cons

  • Matchup feature comes later
B

Feature-Led

Build the matchup matrix and tournament meta advisor first — it's the highest-value new capability. UX polish and code quality improvements come afterwards.

Pros

  • Most exciting feature ships sooner

Cons

  • Risk of matchup data being stale/wrong with no visibility
  • Building on shaky ground
C

Parallel Streams

Quick UX wins (sync status banner, last-refreshed timestamps) ship immediately. Matchup feature is built alongside ongoing code quality improvements woven into each area as it's touched.

Pros

  • Visible progress fast
  • Data transparency arrives before matchup feature

Cons

  • More context-switching
  • Code quality improvements are piecemeal
